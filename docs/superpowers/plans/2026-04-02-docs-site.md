# Getting Started

Splash is a compiled language. You run the Splash compiler to check and build `.splash` files; the compiler transpiles to Go and invokes `go build` for you.

**Prerequisites:** Go 1.21 or later. Verify with `go version`.

---

## Install

Clone the repository and build the compiler:

```bash
git clone https://github.com/zachgraves/gosplash.git
cd gosplash
go build ./cmd/splash/...
```

This produces a `splash` binary in the current directory.

---

## Your First Check

Run the compiler's safety checker on one of the included examples:

```bash
./splash check examples/effects/effects.splash
```

Expected output:

```
examples/effects/effects.splash: ok
```

The checker runs the full pipeline — parse, type check, call graph, all safety passes — and exits zero if no errors are found.

Try a file that uses data classification:

```bash
./splash check examples/data_safety/data_safety.splash
```

Expected: `ok`. This file defines a `User` type with `@sensitive` and `@restricted` fields and verifies that none of the functions violate classification rules.

---

## Inspect Tool Schemas

`splash tools` extracts JSON Schema for every `@tool` function in a file:

```bash
./splash tools examples/agent_tools/agent_tools.splash
```

Expected output:

```json
[
  {
    "name": "search_catalog",
    "description": "Search the catalog for items matching a query.",
    "parameters": {
      "type": "object",
      "properties": {
        "query": { "type": "string" },
        "limit": { "type": "integer" }
      },
      "required": ["query", "limit"]
    },
    "effects": ["DB.read"]
  },
  {
    "name": "get_item",
    "description": "...",
    "parameters": {
      "type": "object",
      "properties": {
        "item_id": { "type": "integer" },
        "category": { "type": "string" }
      },
      "required": ["item_id"]
    },
    "effects": ["DB.read"]
  }

```

The schema is derived entirely from the function signature and `///` doc comments. `category` is `String?` — optional — so it does not appear in `required`.

# Tutorial: Build a Support Agent

This tutorial builds a customer support agent from scratch. You will write a Splash module, hit real compiler errors, and fix them. By the end you have a working agent with effect constraints, data safety, and an approval gate — verified at compile time.

**What you need:** The `splash` binary built from Getting Started. A text editor and a terminal.

---

## 1. Start with a Tool

Create `support.splash`:

```splash
module support

type OrderSummary {
    id:     Int
    status: String
}

@tool
/// Search orders for a customer.
fn search_orders(
    /// The customer ID to search.
    customer_id: Int,
    /// Maximum results to return.
    limit: Int
) needs DB.read -> List<OrderSummary> {
    return db.query(customer_id, limit)
}

fn run_agent(customer_id: Int) needs Agent, DB.read -> String {
    let orders = search_orders(customer_id, 5)
    return "done"
}
```

Check it:

```bash
./splash check support.splash
```

Expected: `support.splash: ok`

Inspect the schema:

```bash
./splash tools support.splash
```

Expected:

```json
[
  {
    "name": "search_orders",
    "description": "Search orders for a customer.",
    "parameters": {
      "type": "object",
      "properties": {
        "customer_id": {
          "type": "integer",
          "description": "The customer ID to search."
        },
        "limit": {
          "type": "integer",
          "description": "Maximum results to return."
        }
      },
      "required": ["customer_id", "limit"]
    },
    "effects": ["DB.read"]
  }
]
```

The schema comes entirely from the type signature and `///` comments. No hand-written JSON.

---

## 2. Add a Customer Type — Hit a Classification Error

Add a `Customer` type with a PII field, and a tool that returns it:

```splash
type Customer {
    id:    Int
    name:  String
    @sensitive
    email: String
}

@tool
/// Get a customer by ID.
fn get_customer(id: Int) needs DB.read -> Customer {
    return db.find(id)
}
```

Check:

```bash
./splash check support.splash
```

The compiler rejects this:

```
support.splash: error: @tool function "get_customer" returns type Customer
which contains @sensitive field(s): email — PII must not flow into agent context
```

`@sensitive` on `email` raises `Customer`'s classification above `@internal`. `@tool` functions cannot return classified types — the agent's context window is not a secure store.

**Fix:** Create a projection type with only public fields, and return that instead:

```splash
type CustomerProfile {
    id:   Int
    name: String
}

@tool
/// Get a customer profile by ID. Does not include contact information.
fn get_customer(id: Int) needs DB.read -> CustomerProfile {
    let c = db.find(id)
    return CustomerProfile { id: c.id, name: c.name }
}
```

Check again: `support.splash: ok`

---

## 3. Add a Dangerous Operation — Hit a Redline Error

Some operations should never be reachable from an agent. Add one:

```splash
@redline(reason: "Account deletion requires CX team sign-off")
fn delete_account(customer_id: Int) needs DB.admin { }
```

This passes on its own — `delete_account` is not reachable from `run_agent` yet. Now make it reachable:

```splash
fn run_agent(customer_id: Int) needs Agent, DB.read, DB.admin -> String {
    let orders = search_orders(customer_id, 5)
    delete_account(customer_id)
    return "done"
}
```

Check:

```bash
./splash check support.splash
```

The compiler reports the full call path:

```
support.splash: error: @redline function "delete_account" is reachable from agent
  run_agent -> delete_account
  reason: Account deletion requires CX team sign-off
```

`@redline` is a compile-time prohibition. There is no configuration that overrides it. The binary cannot be produced while this path exists.

**Fix:** Remove the call to `delete_account` and remove `DB.admin` from `run_agent`:

```splash
fn run_agent(customer_id: Int) needs Agent, DB.read -> String {
    let orders = search_orders(customer_id, 5)
    return "done"
}
```

Check: `support.splash: ok`

---

## 4. Add a Gated Operation

Refunds are consequential but automatable with oversight. Use `@approve`:

```splash
@approve
/// Issue a refund on an order. Requires human approval before executing.
fn refund_order(order_id: Int, reason: String) needs DB.write, Net -> OrderSummary {
    return db.update(order_id, "refunded")
}
```

Check: `support.splash: ok`

See what the compiler generates for the approval gate:

```bash
./splash emit support.splash
```

Look for `refundOrder` in the output. The generated Go signature is `(OrderSummary, error)` — widened from `OrderSummary`. The first line of the body is `splashApprove("refund_order")`. The call site in `run_agent` calls it like a normal function; the gate fires inside `refundOrder`, invisible to the caller.

The default adapter (`StdinApprovalAdapter`) prompts stdin. In production, you swap it before agent startup:

```go
splash.SetApprovalAdapter(&WebhookApproval{URL: secrets.ApprovalWebhook})
```

---

## 5. Constrain the Agent

Add `@sandbox` and `@budget` to the agent entry point to make its effect surface explicit and auditable:

```splash
@sandbox(allow: [DB.read, DB.write, Net])
@budget(max_cost: 0.05, max_calls: 10)
fn run_agent(customer_id: Int) needs Agent, DB.read -> String {
    let orders = search_orders(customer_id, 5)
    return "done"
}
```

Check: `support.splash: ok`

The compiler verified that every function reachable from `run_agent` uses only `DB.read`, `DB.write`, or `Net`. `DB.admin` is not in the allow list — if `delete_account` were reachable, it would fail on two counts (`@redline` and `@sandbox`).

Try adding an effect that violates the sandbox:

```splash
fn log_metric(name: String) needs Metric { }

fn run_agent(customer_id: Int) needs Agent, DB.read, Metric -> String {
    let orders = search_orders(customer_id, 5)
    log_metric("agent.run")
    return "done"
}
```

Check:

```bash
./splash check support.splash
```

Expected error:

```
support.splash: error: @sandbox on "run_agent": reachable function "log_metric"
uses effect Metric not in allow list
```

**Fix:** Either add `Metric` to the allow list, or remove the call. Remove the call for now and revert `run_agent` to the previous version.
